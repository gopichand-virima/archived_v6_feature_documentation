# 1. Executive Architecture Summary

This repository is a React + Vite single-page application (SPA) that serves versioned, AI-generated product documentation derived from source PRDs. Content is never hand-authored: it is generated by calling the Anthropic API against PRD markdown files, validated through a multi-gate pipeline, and deployed to GitHub Pages on every merge to `main`.

**Stack Table:**

| Layer | Technology | Version |
|-------|-----------|---------|
| Framework | React | 18.3.1 |
| Build tool | Vite | 6.3.5 |
| Language | TypeScript (strict mode) | 5.5.3 |
| CSS | Tailwind v4 | 4.1.17 |
| Markdown renderer | react-markdown + rehype-sanitize + rehype-raw + remark-gfm | 6.0.0 (sanitize) |
| AI generation | Anthropic Claude Sonnet 4 | claude-sonnet-4-20250514 |
| Hosting | GitHub Pages | — |
| Package manager | pnpm | 9.15.0 |
| Node minimum | Node.js | 20.0.0 |

**One-paragraph description:** PRDs in the sibling `v6_prds` repository are the source of truth. When PRDs merge to `main` in that repo, a `repository_dispatch` webhook fires the generation pipeline in this repo (`generate-v6feature-docs.yml`). The pipeline calls Claude to classify each PRD and generate typed deliverables (admin guides, user guides, troubleshooting docs, etc.), writes them as plain GFM `.md` files, tracks each output in a `.generation-manifest.{slug}.json`, rebuilds per-module `index.md` TOC files, validates all output with a GREEN/AMBER/RED gate, and creates a branch + PR for human review before any content reaches `main`. On merge, `deploy-pages.yml` runs `pnpm build` (which runs `pnpm sync-toc` as a prebuild hook) and deploys `dist/` to GitHub Pages via the official `actions/deploy-pages` action.

# 2. Current Site Capabilities

1. **AI-powered PRD-to-documentation generation** — Claude Sonnet classifies PRDs and generates typed deliverables per feature.
2. **Content hash-based skip logic** — PRDs whose content is unchanged since the last run are skipped (exit code 3), avoiding redundant API calls.
3. **Multi-deliverable per PRD** — Each PRD can produce up to 4 deliverables (admin guide, user guide, troubleshooting, concept overview, etc.).
4. **Glossary merge** — Generated terms are merged into a shared `glossary.md` file automatically.
5. **Module-level navigation TOC** — Each module has a pipeline-maintained `index.md` rebuilt after every generation run.
6. **Deletion-safe pipeline** — Deleted PRDs have their generated docs, manifests, and empty folders cleaned up; the module index is rebuilt via `rebuild-module-index.ts`.
7. **AFFECTED_MODULES final pass** — Both addition and deletion loops track which modules were touched; `rebuild-module-index.ts` runs for every affected module at the end, ensuring the index always reflects final on-disk state.
8. **Stale entry detection** — `pnpm eval` nav-integrity suite checks that all arrow-ref paths in `index.md` resolve to real files.
9. **GREEN/AMBER/RED validation gate** — 7 checks on generated content gate every PR before merge.
10. **Branch + PR governance** — Every generation run (even GREEN) creates a branch and PR; no direct push to `main`.
11. **Deletion safety limit** — Pipeline hard-stops if deletions exceed 10 files or 25% of total changed files.
12. **Memory and drift detection** — `docs/memory/repo-memory.json` is the canonical AI memory; `pnpm memory:check` runs git-log-based drift detection.
13. **Session hooks** — `.claude/settings.json` blocks `npm`/`yarn`/`npx`, warns on pipeline-owned edits, warns on legacy `/v6_1/` path references.
14. **Evals** — Three eval suites: `coverage` (CRUD section presence), `nav-integrity` (index.md completeness + link resolution), `structure` (folder parity with PRD source).
15. **Static MDX import system** — Per-module import maps in `src/lib/imports/` bundle content at build time.
16. **4-strategy content loading** — Runtime loader tries static imports → MDX bundle → server fetch → registry fallback.
17. **SPA routing with 404 fallback** — `dist/index.html` copied to `dist/404.html` so GitHub Pages returns the SPA on deep-linked URLs.
18. **Rehype-sanitize XSS protection** — All rendered markdown is sanitized; `<script>` and event handlers are stripped.
19. **16 slash commands** — `.claude/commands/` has commands for audit, generate, validate, eval, memory, security review, release readiness, etc.
20. **Local Docker preview** — `nginx.conf` + `Dockerfile` for local smoke testing (not production).
21. **pnpm-only enforcement** — Hook and CI both block npm/yarn.
22. **CODEOWNERS** — Requires devops review for CI files and docs-team review for memory/governance files.

# 3. Frontend / Build / Rendering Architecture

## Build System

- **Vite 6.3.5** with `@vitejs/plugin-react-swc` (SWC-based fast transforms).
- **Base path: `'./'`** (relative). Required so GitHub Pages assets resolve regardless of whether the repo is served at a root domain or a subpath.
- **Custom plugins in `vite.config.ts`:**
  - `copyContentPlugin` — copies `src/pages/content/` to `dist/content/` at bundle time (content served statically).
  - `copyAssetsPlugin` — copies `src/assets/images/` to `dist/assets/images/`.
  - `serveAssetsPlugin` — dev server middleware for `/assets/images/*` and `/content/*` paths.
- **`prebuild` hook** in `package.json` runs `pnpm sync-toc` before every build.
- **Build target:** `esnext`, output to `dist/`.

## Frontend Stack

- **React 18.3.1** with Radix UI component primitives.
- **Tailwind v4** — `src/index.css` is a pre-compiled artifact; new utility classes not in the original scan must be back-filled in `src/styles/globals.css`.
- **Routing:** Path-based SPA routing (pushState). URL pattern: `/{version}/{module}/{section}/{page}`.
- **Left sidebar:** `ResizableSidebar` with `NavigationMenu` — initial 288px, min 200px, max 500px; hidden below `lg` (1024px).
- **Right TOC rail:** `TableOfContents` component — sticky, auto-extracts `h2/h3/h4` headings via DOM; hidden below `xl` (1280px); width 256px.
- **Dark mode:** `next-themes` provider; all dark variants live in component class strings.

## Rendering Pipeline

```
.md file fetched from /content/{version}/{module}/{feature}/{file}.md
         ↓
contentLoader.ts (4-strategy priority chain)
         ↓
MDXRenderer.tsx
  ├─ react-markdown
  ├─ remark-gfm (GFM tables, strikethrough, etc.)
  ├─ rehype-raw (allow raw HTML passthrough)
  └─ rehype-sanitize (XSS: blocks <script>, event handlers; allows className/style)
         ↓
DocumentationContent.tsx (TOC rail + breadcrumbs + article column)
```

## Content Loading Strategy (Priority Order)

| Priority | Strategy | Source | Speed |
|----------|---------|--------|-------|
| 1 | Static MDX imports | `src/lib/imports/*.ts` bundled at build time | Fastest |
| 2 | MDX bundle | `import.meta.glob` dynamic import | Fast |
| 3 | Fetch from server | `GET /content/6_1/{module}/{feature}/{file}.md` | Standard |
| 4 | Registry placeholder | Synthetic content registry | Fallback only |

Note: As of the current state, Strategy 3 (server fetch) is the primary active path; static imports in Strategy 1 are present but the merge/priority logic effectively routes most traffic through fetch.

# 4. Content Model and Information Architecture

## File Hierarchy

```
src/pages/content/
  6_1/                          ← active version
    {module}/                   ← e.g., admin, itsm, cmdb
      index.md                  ← pipeline-maintained module TOC
      {feature_slug}/           ← e.g., ad_configuration
        {deliverable}.md        ← e.g., ad-configuration-module-administration.md
        .generation-manifest.{slug}.json
    glossary.md                 ← shared glossary (merged at generation time)
  _GENERATION_SUMMARY.md        ← pipeline summary (committed with docs)
```

## Module Index Format

```markdown
## {Module Title}

### {Feature Name}
- {Deliverable Label} → {module}/{feature_slug}/{filename}.md

### {Another Feature Name}
- {Deliverable Label} → {module}/{feature_slug}/{filename}.md
```

Rules:
- Plain `.md` — not `.mdx`.
- Pipeline-maintained — rebuilt by `generate-feature-doc.ts` after additions; `rebuild-module-index.ts` for final pass after both additions and deletions.
- Alphabetically sorted by feature name.
- Paths relative to version root (e.g., `admin/sla/sla-management-user-guide.md`).

## Generation Manifest Schema

```json
{
  "sourcePath": "src/pages/6_1/{module}/{feature}/{feature}.md",
  "outputDir": "src/pages/content/6_1/{module}/{feature}",
  "prdTitle": "...",
  "prdVersion": "1.0",
  "prdStatus": "Draft",
  "prdFocusArea": "...",
  "suite": "6 1",
  "module": "{module} > {feature}",
  "featureName": "...",
  "featureSummary": "...",
  "model": "claude-sonnet-4-20250514",
  "generatedAt": "ISO-8601",
  "prdContentHash": "sha256-hex",
  "prdNormalizedHash": "sha256-hex",
  "deliverables": [
    {
      "type": "admin-guide",
      "outputFile": "src/pages/content/6_1/{module}/{feature}/{file}.md",
      "tokenUsage": { "input": 6511, "output": 1820 },
      "navTitle": "Human-readable H1-derived title"
    }
  ],
  "glossaryTermsAdded": 9
}
```

## Naming Conventions

| Artifact | Convention | Example |
|---------|-----------|---------|
| Feature folder | `lowercase_underscore` | `ad_configuration` |
| Deliverable filename | `kebab-case.md` | `ad-configuration-module-administration.md` |
| Component files | `PascalCase.tsx` | `DocumentationLayout.tsx` |

| Script files | `kebab-case.ts` or `camelCase.ts` | `rebuild-module-index.ts` |
| Manifest | `.generation-manifest.{slug}.json` | `.generation-manifest.ad_configuration.json` |

## Deliverable Types

| Type | Description |
|------|-------------|
| `user-guide` | End-user how-to |
| `admin-guide` | Administrator configuration guide |
| `concept-overview` | Background explanation |
| `troubleshooting` | Problem resolution guide |

# 5. Generation Pipeline and Artifact Lifecycle

## High-Level Flow

```
v6_prds: PRD merged to main
    ↓  notify-feature-docs.yml fires repository_dispatch
generate-v6feature-docs.yml
    ├── Step: Resolve event inputs (workflow_dispatch vs repository_dispatch)
    ├── Step: Checkout v6_feature_documentation
    ├── Step: Checkout v6_prds at _prd-source/
    ├── Step: Install pnpm + dependencies
    ├── Step: Preflight validation (validate changes payload, path constraints)
    ├── Step: Generate feature documentation
    │       ├── Loop: Additions + modifications
    │       │     └── generate-feature-doc.ts (4-stage AI pipeline, exit codes 0–4)
    │       ├── Loop: Deletions (rm deliverables, manifest, empty folder)
    │       └── Final pass: rebuild index for every AFFECTED_MODULE
    ├── Step: Create synthetic manifests for module index.md files
    ├── Step: Validate generated output (no .mdx in versioned dirs)
    ├── Step: Run validation gate (validate-generation.ts, GREEN/AMBER/RED)
    ├── Step: Build PR metadata from manifests
    ├── Step: Create branch + PR (ALL gates)
    └── Step: Post job summary
```

## generate-feature-doc.ts: 4-Stage Script

| Stage | Name | Action |
|-------|------|--------|
| 0 | Pre-Generation Validation | Checks existing docs, detects change type, gates proceed/skip |
| 1 | Classification | Anthropic API: analyze PRD, determine deliverable types, extract glossary terms |
| 2 | Per-Deliverable Generation | Anthropic API: generate each .md deliverable |
| 3 | Glossary Merge + Manifest | Update glossary.md, write `.generation-manifest.{slug}.json` |

Exit codes:
- `0` = success
- `1` = error (generation failed)
- `2` = skipped (empty/dummy/placeholder PRD)
- `3` = skipped (no changes — content hash matches last run)
- `4` = skipped (non-functional change — metadata only)

## AFFECTED_MODULES Tracking Logic

The workflow maintains an `AFFECTED_MODULES` variable (newline-delimited module dir paths). Both the additions loop and the deletions loop append to this set using a deduplication guard:

```bash
if [[ -n "$MODULE_DIR" ]] && ! printf '%b' "$AFFECTED_MODULES" | grep -qxF "$MODULE_DIR"; then
  AFFECTED_MODULES="${AFFECTED_MODULES}${MODULE_DIR}\n"
fi
```

After both loops complete, the final pass iterates this set and runs `rebuild-module-index.ts --module-dir` for each entry.

## Edge Case Coverage

| Scenario | Behavior |
|---------|---------|
| Add-only batch | Normal generation loop runs; each module added to AFFECTED_MODULES; final pass rebuilds index |
| Modify-only batch | Content hash match = exit code 3 (skip); no regeneration |
| Delete-only batch | Deletion loop cleans deliverables + manifest + empty dir; module added to AFFECTED_MODULES; final pass rebuilds index |
| Add + delete, same module | Both loops add same module to AFFECTED_MODULES (deduplicated); final pass runs once for that module |
| Add + delete, different modules | Both modules tracked independently; final pass rebuilds each |
| Zero-deliverable features | `rebuild-module-index.ts` warns and skips (guard at line 165) |
| Malformed manifest (missing featureName) | `rebuild-module-index.ts` warns and skips (guard at line 152) |
| Stale index.md after deletions | Fixed by AFFECTED_MODULES final pass — index always reflects final on-disk manifest state |
| PRD source file not found | Logged as FAILED; added to failed_files output; pipeline continues with remaining PRDs |
| All PRDs skipped | `has_generated=false`; no PR created; no validation gate runs |

## Deletion Safety Gate

Environment variables (configurable per run):
- `DELETION_MAX_FILES=10` — RED if more than 10 files deleted.
- `DELETION_MAX_PERCENT=25` — RED if deletions exceed 25% of total changed files.

The `deletion_max_percent` can be raised via `workflow_dispatch` input for planned major restructures.

## Synthetic Manifests for Module Index Files

Module `index.md` files are pipeline-maintained TOCs — they are not per-PRD outputs. The pipeline creates lightweight synthetic manifests (`.generation-manifest.{module}-index.json`) so `validate-generation.ts checkManifestTraceability` treats them as known pipeline-owned files rather than untracked content. It also prepends an H1 heading to index.md files (which start with `##`) to satisfy `checkH1Sanity`.

# 6. Navigation / Index / Manifest Architecture

## Data Flow

```
src/pages/content/6_1/{module}/index.md  (source of truth for nav)
         ↓  pnpm sync-toc (sync-toc-from-index.ts)
src/data/navigationData.ts               (auto-generated — do not hand-edit)
src/utils/indexContentMap.ts             (auto-generated — do not hand-edit)
         ↓  React app reads at runtime
NavigationMenu.tsx (left sidebar tree)
```

## sync-toc-from-index.ts

- Reads all `src/pages/content/{version}/*/index.md` files.
- Parses `##` as module title, `###` as section/feature name, `- Label → path.md` as page entries.
- Writes `navigationData.ts` (structured navigation tree) and `indexContentMap.ts` (raw string map for TOC rendering).
- Active versions array: `['6_1']` — 6.1.1, 5.13, NextGen defined but excluded.
- Module ID to variable name mapping: `MODULE_VAR_MAP` in the script.

## rebuild-module-index.ts

- Called per affected module after both addition and deletion loops.
- Scans all subdirectories of the module dir for `.generation-manifest.*.json` files.
- Reads `featureName` and `deliverables[].outputFile` + `navTitle` from each manifest.
- Sorts features alphabetically by `featureName`.
- Writes `index.md` in the pipeline format.
- If no manifests found: writes a minimal stub with a notice (does not fail).

## Navigation Hierarchy

```
Module (## heading in index.md → module title)
  Section/Feature (### heading → section title)
    Page (- label → relative/path.md)
```

URL pattern: `/{version}/{module}/{section}/{page}`

# 7. Evals, Quality Gates, and Acceptance Criteria

## pnpm eval Command

Runs `scripts/evals/run-evals.ts`. Executes four suites: `structure`, `coverage`, `nav-integrity`, `compliance`.

| Suite | What It Checks |
|-------|---------------|
| `structure` | Folder parity with PRD source: every PRD folder with a `{slug}.md` must have a matching generated doc. Requires `v6_prds` repo present at `C:\github\v6_prds`. |
| `coverage` | Every non-index `.md` file contains add/create, edit/update, and delete/remove keywords. |
| `nav-integrity` | Every module has an `index.md`; every arrow-ref path in `index.md` resolves to a real file; every feature subdirectory has at least one `.md` file. |
| `compliance` | `pnpm-lock.yaml` exists; `package-lock.json` absent; `packageManager` field starts with `pnpm`; no forbidden root JS files; `CLAUDE.md` present; `.claude/commands/` present. |

## GREEN / AMBER / RED Interpretation

| Gate | Exit Code | Meaning | Action |
|------|-----------|---------|--------|
| GREEN | 0 | All checks pass | Safe to merge |
| AMBER | 2 | Warnings — low-risk issues | Human review required |
| RED | 1 | High-risk issues detected | Fix before merge; workflow fails |

## Tolerated AMBER Conditions

- Troubleshooting docs missing CRUD sections — by design (troubleshooting guides do not have add/edit/delete sections). These show as AMBER `coverage` findings and are pre-existing and expected.
- `v6_prds` repo absent on CI runner — `structure` suite emits AMBER and skips parity check.

## Eval vs Other Gates

| Gate | Tool | What It Covers | Blocks Merge? |
|------|------|----------------|---------------|
| `pnpm typecheck` | `tsc` | TypeScript types | Yes |
| `pnpm check:security` | `verify-security.ts` | Hardcoded secrets | Yes |
| `pnpm check:setup` | `verify-setup.ts` | Repo structure | Yes |
| `pnpm check:paths` | `verify-paths.ts` | Path contracts | Yes |
| `pnpm eval` | `run-evals.ts` | Doc quality + nav + parity | AMBER = no; RED = yes |
| `pnpm memory:check` | `check-memory-drift.ts` | Memory drift | Warning only (`continue-on-error: true`) |
| `pnpm build` | Vite | Full build | Yes |

## validate-generation.ts: 7 Checks

| # | Check | Pass | Warn | Fail |
|---|-------|------|------|------|
| 1 | h1-sanity | All `.md` start with `# ` | 1 file missing H1 | 2+ files missing H1 |
| 2 | min-file-size | All files >= 200 bytes | Any file < 200 bytes | — |
| 3 | filename-rules | All lowercase-hyphen, no `.mdx`, no PRD tokens | Any violation | — |
| 4 | manifest-traceability | All content dirs at depth > 0 have a manifest | Legacy manifest only | Missing manifest |
| 5 | semantic-smoke-check | No banned tokens (TODO, REPLACE_ME, sprint, stakeholder, etc.) | — | Any banned token found |
| 6 | deletion-safety | Deletions within limits | — | Exceeds DELETION_MAX_FILES or DELETION_MAX_PERCENT |
| 7 | git-diff-sanity | Changes tracked by git | — | Unexpected state |

# 8. Validation, Drift Checks, Memory Contracts, and Repo Contracts

## Memory System

- **`docs/memory/repo-memory.json`** — canonical AI memory for the repository. Versioned (`memory_version`), timestamped (`generated_at`).
- **`scripts/memory/check-memory-drift.ts`** — two strategies: (1) VALUE CROSS-CHECK reads actual values from code files and compares against what memory records; (2) FILE-CHANGE TRACK uses `git log` since memory's `generated_at` to find which sections need updating.
- **`scripts/memory/change-routing-map.ts`** — maps changed file patterns to memory sections that need updating.
- **`scripts/memory/update-memory.ts`** — updates memory to current state.
- **Exit codes:** 0 = GREEN, 1 = AMBER, 2 = RED.
- Memory check runs in CI with `continue-on-error: true` (warning only).

## Repo Contracts

| Contract | Value |
|---------|-------|
| Package manager | pnpm 9.15.0 only |
| Node minimum | 20.0.0 |
| Language | TypeScript only (`.ts`/`.tsx`); one documented JS exception: `public/validate-fix.js` |
| Build output | `dist/` — never `build/` |
| Base path | `'./'` — relative, required for GitHub Pages |
| Content ownership | `src/pages/content/` is pipeline-owned |
| Index.md ownership | Module index files are pipeline-maintained |
| Strict TypeScript | `strict: true`; no `any` without justification comment |
| Shared types | `src/types/` |

## Claude Code Session Hooks (`.claude/settings.json`)

| Hook | Trigger | Action |
|------|---------|--------|
| PreToolUse (Bash) | Any `npm `, `yarn `, `npx ` command | Blocks with exit 2, instructs to use `pnpm` |
| PreToolUse (Edit/Write) | Edit to `src/pages/content/` path | Warning: pipeline-owned, manual edits will be overwritten |
| PreToolUse (Edit/Write) | Content references `/v6_1/` | Warning: legacy path, correct is `/6_1/` |
| Stop | Session ends | Reminder to run `pnpm memory:check`, `pnpm validate`, `pnpm build` |

## Prohibited Paths

- `src/content/` — old path, never re-create.
- `build/` — wrong output dir, always use `dist/`.
- `src/pages/v6_prds/` — PRDs belong in sibling repo.

# 9. Governance, Branching, PR Flow, and GitHub Settings Dependencies

## Branch + PR Policy

- **All three gate outcomes** (GREEN, AMBER, RED) create a branch and PR. No auto-merge.
- Branch naming convention: `autofeature-featuredocs-6_1-{DDMMYYYY}-{HHMMSS}` (no slashes).
- Example: `autofeature-featuredocs-6_1-03042026-143022`.
- **Human-only merge** — no workflow has push-to-main permission for content branches.
- RED gate: creates the PR but also exits with code 1, failing the workflow. The PR exists so reviewers can inspect the issue.

## Required GitHub Settings (Cannot Be Configured in Repo Files)

| Setting | Location | Required Value |
|---------|---------|---------------|
| Pages source | Settings → Pages → Source | GitHub Actions (not branch-based) |
| PR creation permission | Settings → Actions → General → Workflow permissions | "Allow GitHub Actions to create and approve pull requests" must be checked |
| Secret: ANTHROPIC_API_KEY | Settings → Secrets and variables → Actions | Anthropic API key for generation |
| Secret: PRD_REPO_TOKEN | Settings → Secrets and variables → Actions | PAT with read access to `v6_prds` repo |

## CODEOWNERS

- `Dockerfile`, `nginx.conf`, workflow files → devops team review required.
- `CLAUDE.md`, `docs/memory/`, `scripts/checks/` → docs-team review required.

## Workflow Concurrency

- `generate-v6feature-docs.yml`: `group: generate-v6feature-docs`, `cancel-in-progress: false` — queues, never cancels.
- `deploy-pages.yml`: `group: pages-deploy`, `cancel-in-progress: false` — never cancels in-flight deploy.

# 10. Cross-Repo Trigger and Publish Topology

```
virima-products/v6_prds (source repo)
    └── .github/workflows/notify-feature-docs.yml (or trigger-v6feature-docs.yml)
            ↓  repository_dispatch event type: "generate-feature-docs"
            ↓  client_payload: { changes, source_commit, source_author, source_ref, full_rebuild }
virima-products/v6_feature_documentation (this repo)
    └── .github/workflows/generate-v6feature-docs.yml
            ↓  checks out v6_prds at _prd-source/ using PRD_REPO_TOKEN
            ↓  runs generation pipeline
            ↓  creates branch + PR
            ↓  (on merge to main) deploy-pages.yml triggers
            ↓  GitHub Pages
```

**Key details:**
- The `PRD_REPO_TOKEN` secret must be a PAT with `contents: read` on the `v6_prds` repo.
- The `v6_prds` sender workflow fires on PRD merges to `main` and sends the list of changed files as JSON.
- `full_rebuild: true` can be sent to regenerate all PRDs (uses `--force` flag, bypasses content hash check).
- `skip_generation: true` (workflow_dispatch only) — skips Claude API calls, validates existing docs only.

# 11. Deployment, Runtime Behavior, and GitHub Pages Model

## Production Deployment

| Trigger | Workflow | Action |
|---------|---------|--------|
| Push to `main` | `deploy-pages.yml` | Build + deploy to GitHub Pages |
| PRD change merge | `generate-v6feature-docs.yml` | Generates docs → branch + PR → deploy on merge |
| Feature branch push / PR | `ci.yml` | Validate + build (no deploy) |

## Build Steps (deploy-pages.yml)

1. Checkout.
2. Setup pnpm + Node.js 20 with pnpm cache.
3. `pnpm install --frozen-lockfile`.
4. `pnpm build` — runs `pnpm sync-toc` (prebuild), then Vite build to `dist/`.
5. `cp dist/index.html dist/404.html` — SPA fallback for deep links.
6. `actions/configure-pages@v5`.
7. `actions/upload-pages-artifact@v3` — uploads `dist/`.
8. `actions/deploy-pages@v4`.

**Live URL:** `https://virima-products.github.io/v6_feature_documentation/`

## Runtime Behavior

- The SPA handles all routing client-side.
- Content files are served statically from `dist/content/` (copied by `copyContentPlugin`).
- The fetch strategy requests markdown files at `/content/6_1/{module}/{feature}/{file}.md` (relative base path resolves correctly).
- `index.html` is duplicated as `404.html` so GitHub Pages returns the SPA on unrecognized paths.

## Local Docker Preview (Optional)

```bash
docker compose up --build
# Site at: http://localhost:8080/v6_feature_documentation/
```
Uses `nginxinc/nginx-unprivileged:alpine`, nginx runs as uid 101, port 8080. Not used in production CI/CD.

# 12. Security, Dependency Risk, and Sanitization Model

## Architecture Security Model

This is a **public-facing static documentation site** with no user accounts, no backend, and no server-side state.

| Layer | Mechanism |
|-------|----------|
| TLS / HTTPS | GitHub Pages CDN |
| DDoS / rate limiting | Edge (not in-app) |
| Authentication | None required |
| Content rendering XSS protection | `rehype-sanitize` in `MDXRenderer.tsx` |
| HTTP security headers | `nginx.conf` (local Docker only) |
| Secret scanning | `pnpm check:security` (CI) |
| Supply-chain governance | `.github/CODEOWNERS` |
| Container | `nginxinc/nginx-unprivileged:alpine` (local only) |

## rehype-sanitize Configuration

- Extends `defaultSchema` (hast-util-sanitize).
- **Allows:** `className`, `style` on all elements; `className` on `code`/`span`/`div` for Tailwind and syntax highlighter.
- **Blocks:** All event handlers (`onclick`, `onerror`, etc.), `<script>`, `<iframe>`, and any other elements not in `defaultSchema`.
- This is the primary XSS mitigation for AI-generated content rendered in the browser.

## Secret Scanning Patterns (verify-security.ts)

| Pattern | What It Catches |
|---------|----------------|
| `/sk-[a-zA-Z0-9]{20,}/` | OpenAI API keys |
| `/sk-ant-[a-zA-Z0-9\-_]{20,}/` | Anthropic API keys |
| `/AKIA[0-9A-Z]{16}/` | AWS Access Key IDs |
| `/ghp_[A-Za-z0-9]{36}/` | GitHub classic PATs |
| `/github_pat_[A-Za-z0-9_]{22,}/` | GitHub fine-grained PATs |

Skips: `node_modules`, `.git`, `build`, `dist`, `.env.local`, `verify-security.ts` itself.
Scans: `.ts`, `.tsx`, `.js`, `.jsx`, `.mjs`, `.json`.

## VITE_* Bundle Exposure Risk (Known Limitation)

Any `VITE_*` environment variable is compiled into the JavaScript bundle and visible in browser DevTools. API keys (OpenAI, Anthropic) MUST NOT be prefixed with `VITE_`. The `ANTHROPIC_API_KEY` is used server-side in the CI workflow only, never in the client bundle.

**Required future work:** Proxy API calls through a backend-for-frontend (BFF) before exposing to untrusted public networks.

## Dependency Vulnerabilities

As of the last audit, Dependabot reports approximately 10 open vulnerabilities:
- 1 high severity: `lodash` transitive dependency.
- 5 moderate severity: `vite`/`mdast` related packages.
- 4 low severity: `vite` related packages.

These are tracked via Dependabot alerts and require periodic triage.

# 13. Performance, UX, Accessibility, and Search Notes

## Performance

- Content files served as static assets from `dist/content/` — no server-side rendering.
- Static import system (`src/lib/imports/`) bundles content at build time for fastest load (Strategy 1).
- Image optimization: `OptimizedImage` component with `IntersectionObserver` lazy loading; `priority=true` bypasses lazy loading for above-fold images.
- In-memory content cache in `contentLoader.ts` (`Map<string, string>`).
- Vite's `esnext` target = minimal polyfills, smaller bundles.

## UX

- Left sidebar: resizable (200px–500px), persistent across navigation.
- Right TOC: `IntersectionObserver` with `-80px 0px -80% 0px` rootMargin for accurate active heading tracking.
- Dark mode: `next-themes` provider, full dark variant coverage in component class strings.
- Error states: `ContentNotAvailable` component shows file path, error details, context.
- Empty state: module landing shows "Select a topic from the sidebar" prompt.
- SPA 404 fallback ensures deep links work on GitHub Pages.

## Accessibility

**Not implemented.** No ARIA audit, no keyboard navigation testing, no screen reader testing, no color contrast compliance testing has been done. Radix UI primitives provide baseline accessibility for interactive components, but no systematic accessibility review has been completed.

## Search

No search implementation is currently active. The dependency `cmdk` (command palette) is listed in `package.json` and there is a `src/lib/search/` directory, but no working search feature is deployed. A `VITE_*` API key approach for AI search would require BFF proxying before production use.

# 18. Failure Modes, Recovery Paths, and Known Sharp Edges

| Failure | Symptoms | Recovery |
|---------|---------|---------|
| Generation fails mid-batch | Some PRDs processed, others failed; `failed_files` output populated; partial manifest state | Identify failed PRDs from Actions summary; re-run pipeline with those PRDs; merge partial PR manually if safe |
| PR creation blocked by repo permissions | Workflow logs: "PR creation failed — branch pushed, create PR manually" | Enable "Allow GitHub Actions to create and approve pull requests" in repo settings; branch still exists on remote |
| Stale index.md after deletions | Nav shows entries for deleted features; nav-integrity eval shows AMBER stale entries | Run `pnpm rebuild-index -- --module-dir src/pages/content/6_1/{module}`; then `pnpm sync-toc` |
| navigationData.ts out of sync | Nav shows wrong structure or missing modules | Run `pnpm sync-toc`; commit the result |
| Content hash collision (skip when should regenerate) | Stale docs despite PRD changes | Run with `full_rebuild: true` via workflow_dispatch |
| GitHub Pages not deployed | 404 on live URL; deploy-pages.yml never ran | Check Settings → Pages → Source is "GitHub Actions" (not branch-based) |
| Tailwind class missing (v4) | UI element unstyled | Back-fill class in `src/styles/globals.css` |
| pnpm version mismatch | `ERR_INVALID_THIS` or lockfile errors | Ensure Node 20+ and pnpm 9.15.0 exactly |
| Vite base path wrong | Assets 404 on GitHub Pages | Base path must be `'./'` — do not set to absolute path |
| Memory drift RED | `check-memory-drift` fails with value mismatch | Run `/sync-repo-memory` command or `pnpm memory:update` |
| Module index written by generate-feature-doc.ts (not rebuild-module-index.ts) | H1 heading missing from index.md | The synthetic manifest step in the workflow prepends H1 — check workflow logs |
| Dependabot lodash vulnerability | Security alert | Update transitive dependency via `pnpm update --recursive` |

# 20. New Docsite Blueprint

Step-by-step guide to reproduce this architecture for a different product:

**Prerequisites:**
- Two GitHub repos: `{org}/{product}-prds` (source) and `{org}/{product}-feature-docs` (this pattern).
- GitHub Pages enabled (Settings → Pages → Source → GitHub Actions).
- GitHub Actions PR creation permission enabled.
- Repository secrets: `ANTHROPIC_API_KEY`, `PRD_REPO_TOKEN`.
- Node 20+, pnpm 9.15.0.

**Step 1: Bootstrap the repo**
```bash
pnpm create vite {product}-feature-docs --template react-swc-ts
cd {product}-feature-docs
pnpm install
```

**Step 2: Copy and adapt the nervous-system files**
- Copy `vite.config.ts` — update base path if needed; keep copy plugins.
- Copy `package.json` — update `name`; keep all scripts and pnpm enforcement.
- Adapt `CLAUDE.md` — update repo name, live URL, version, paths.
- Copy `.claude/settings.json` — update pipeline path guards.

**Step 3: Set up the content structure**
```
src/pages/content/{version}/         ← one dir per product version
  {module}/                          ← mirrors PRD repo structure
    index.md                         ← will be pipeline-generated
```

**Step 4: Set up scripts**
- Copy `scripts/generate-feature-doc.ts` — adapt prompts for your product domain.
- Copy `scripts/path-mapping.ts` — update version normalization if needed.
- Copy `scripts/validate-generation.ts` — works generically.
- Copy `scripts/rebuild-module-index.ts` — works generically.
- Copy `scripts/sync-toc-from-index.ts` — update `MODULE_VAR_MAP` and `VERSIONS`.
- Copy `scripts/evals/run-evals.ts` — update `PRD_DIR` path.
- Copy `scripts/checks/` — adapt path patterns.
- Copy `scripts/memory/` — adapt `VALUE_CHECKS`.

**Step 5: Set up CI/CD workflows**
- Copy `.github/workflows/deploy-pages.yml` — no changes needed.
- Copy `.github/workflows/ci.yml` — no changes needed.
- Adapt `.github/workflows/generate-v6feature-docs.yml` — update repo names, secrets, content paths.
- Set up trigger workflow in PRD repo.

**Step 6: Set up the React app**
- Copy `src/components/MDXRenderer.tsx` — do not weaken rehype-sanitize.
- Copy `src/lib/content/contentLoader.ts` — adapt import maps.
- Set up navigation components reading from `src/data/navigationData.ts`.

**Step 7: Initialize memory**
- Create `docs/memory/repo-memory.json` reflecting your new repo's contracts.
- Adapt `scripts/memory/change-routing-map.ts`.

**Step 8: First generation run**
```bash
pnpm install
pnpm validate       # should pass on clean repo
# Trigger workflow_dispatch with full_rebuild=true to generate all content
# Review and merge the generated PR
pnpm build          # verify build works
```

**Step 9: Ongoing governance**
- Run `pnpm validate` before every commit.
- Run `pnpm memory:check` after architecture changes.
- Use `pnpm eval` after content changes.
- Never hand-edit files in `src/pages/content/`.
- Never hand-edit `navigationData.ts` or `indexContentMap.ts`.
